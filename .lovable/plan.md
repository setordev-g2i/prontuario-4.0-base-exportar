## Plano: Cadastro completo de Médicos Responsáveis

### 1. Banco de dados e seeds

Criar uma migração SQL com:

- Tabelas auxiliares solicitadas, cada uma com `id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` e coluna `descricao` ou `nome` conforme sua regra.
- Tabela principal `medico_responsaveis` com todos os campos informados, `created_at`, `updated_at` e todas as foreign keys exatas.
- Tabelas filhas:
  - `medico_responsaveis_cbo`
  - `medico_responsaveis_sus_exportacao`
- Índices para os principais relacionamentos e busca por `nome`, `cpf` e `crm`.
- Trigger para manter `updated_at` atualizado automaticamente.
- Seed inicial para todas as tabelas auxiliares com os valores que você listou.

### 2. Segurança e acesso ao backend

Como o projeto agora usa backend integrado, estruturar o acesso com políticas de segurança para o CRUD funcionar corretamente:

- Habilitar RLS nas tabelas criadas.
- Criar políticas iniciais para usuários autenticados poderem listar, inserir, editar e desativar registros.
- Como você pediu exclusão, vou implementar exclusão lógica no cadastro principal para seguir a regra do projeto. Para isso, adicionarei um campo de status de ativo/inativo sem quebrar sua modelagem principal.

### 3. Estrutura de rotas e navegação

Criar rotas reais do TanStack Start antes de linkar no menu:

- `/configuracoes`
- `/configuracoes/profissionais`
- `/configuracoes/profissionais/medico-responsaveis`

Atualizar a sidebar em `Configurações > Profissionais` para incluir o submenu **Médicos Responsáveis**.

### 4. Camada de tipos e serviços

Criar uma camada reutilizável para o módulo:

- Tipos TypeScript do formulário, filtros, listagem e entidades relacionadas.
- Helpers para máscara/normalização de CPF.
- Serviços de backend para:
  - listar médicos
  - buscar por id
  - salvar cadastro
  - editar cadastro
  - desativar/excluir logicamente
  - gerenciar CBOs vinculados
  - gerenciar exportações SUS
  - listar opções dos selects
  - criar rapidamente registros simples nas tabelas auxiliares quando estiverem vazias

### 5. Página de listagem

Criar a tela **Cadastro de Médicos Responsáveis** com:

- Cabeçalho com título.
- Botão **Novo Médico Responsável**.
- Busca por nome, CPF ou Tipo de Conselho
- Tabela responsiva com colunas:
  - Nome
  - CPF
  - CRM
  - Situação
  - Tipo de Conselho
  - Unidade
  - Ações
- Botão de ações no padrão do projeto: **Opções** com seta para baixo, abrindo no hover, contendo no mínimo:
  - Visualizar
  - Editar
  - Desativar

### 6. Formulário com abas

Criar formulário em modal largo ou página dedicada, usando componentes já existentes, com 5 abas:

- **Dados principais**
- **Dados complementares**
- **Financeiro / Apuração**
- **CBOs vinculados**
- **Exportação SUS**

Implementação prevista:

- `react-hook-form` + `zod` para validação.
- Selects carregados das tabelas relacionadas.
- Se alguma tabela auxiliar estiver vazia, exibir ação de cadastro rápido no próprio fluxo.
- CPF com máscara `000.000.000-00`.
- Campos obrigatórios: nome, cpf, crm, situação, tipo de conselho, unidade.
- Mensagens de erro por campo e bloqueio do envio quando inválido.

### 7. Abas filhas

#### CBOs vinculados

- Seleção múltipla controlada via tabela `medico_responsaveis_cbo`.
- Adicionar/remover vínculos no formulário de edição.

#### Exportação SUS

- Listar datas de exportação já vinculadas.
- Botão para adicionar nova data.
- Remover registros de exportação com confirmação.

### 8. Feedbacks e UX

Adicionar:

- Toasts de sucesso e erro com `sonner`.
- Confirmação antes de desativar médico e antes de remover exportação/CBO quando aplicável.
- Estados de carregamento, vazio e erro.
- Layout responsivo para desktop e tablet.

## Detalhes técnicos

- Não há tabelas públicas no backend atualmente; a migração criará toda a base do módulo do zero.
- O projeto já possui `react-hook-form`, `zod`, `tabs`, `dialog`, `table`, `dropdown-menu`, `select` e cliente Supabase prontos para uso.
